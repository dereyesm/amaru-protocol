> *"UDP and TCP coexist because different data has different needs.
> A heartbeat does not need a handshake. A contract does not tolerate
> silence. The protocol must know the difference — and act accordingly."*

## 1. Abstract

This specification defines two transport modes for HERMES messages: **Datagram Mode (DGM)** and **Reliable Mode (REL)**. DGM provides fire-and-forget delivery suitable for informational messages. REL provides tracked delivery with resolution confirmation and escalation on expiry, suitable for task-oriented messages.

The transport mode is determined by the message `type` field (ARC-5322). No new fields are added to the message schema. Correlation between requests and responses uses a payload convention within the existing `msg` field, preserving full backward compatibility with Phase 0 implementations.

This specification extends ARC-0793 (Reliable Transport) from session-level reliability to **message-level reliability**.

## 2. Motivation

ARC-0793 establishes session-level reliability through SYN/FIN handshakes and at-least-once delivery. However, it treats all messages identically at the transport layer. A `state` announcement and a `dispatch` task assignment receive the same guarantees: both expire silently when their TTL runs out.

This creates the **silent drop problem**: a namespace sends a `dispatch` or `request`, the recipient ACKs it (marking "I saw it"), but never completes the requested action. When the message's TTL expires, it is archived without notification. The requesting namespace has no way to know whether the task was completed, forgotten, or failed.

In networking terms: HERMES Phase 0 provides only UDP semantics. Some messages need TCP semantics — guaranteed delivery with completion tracking and failure notification.

### 2.1 Real-World Evidence

In the reference deployment (Clan Momosho D.), the bus accumulated 77 messages in 5 days. Analysis revealed:

- `dispatch` messages (task assignments) that expired with ACK but without resolution
- `data_cross` messages (inter-dimensional data transfers) with zero ACKs
- `request` messages whose responses were never formally linked to the originals

The current `archive_expired` function treats all expired messages equally, regardless of whether they represent completed work or dropped tasks.

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Datagram Mode (DGM)** | Fire-and-forget transport. Messages are informational. Silent archival on TTL expiry. |
| **Reliable Mode (REL)** | Tracked transport. Messages represent tasks or data transfers. Escalation on unresolved expiry. |
| **Transport Mode** | DGM or REL, determined by the message `type` field. |
| **Correlation ID (CID)** | A token embedded in the `msg` field that links a request to its response. Format: `[CID:token]`. |
| **Resolution Tag (RE)** | A token in a companion message's `msg` field that references a CID. Format: `[RE:token]`. |
| **Resolution** | A companion message with `[RE:token]` confirming completion (or failure) of a REL message. |
| **Escalation** | An alert generated when a REL message expires without resolution. |
| **Companion Message** | A message whose `msg` field contains `[RE:token]` referencing an existing `[CID:token]`. |

## 4. Transport Mode Classification

### 4.1 Classification Rules

The transport mode is determined by the message `type` field. No explicit mode field is required.

| Type | Mode | Rationale |
|------|------|-----------|
| `state` | DGM | Informational broadcast. No action expected. |
| `event` | DGM | Records an occurrence. No response expected. |
| `dojo_event` | DGM | Evolution milestone. Informational. |
| `alert` | DGM | Urgent information. Requires attention, not formal response. |
| `request` | REL | Explicitly asks for something. Expects a response. |
| `dispatch` | REL | Assigns a task. Expects completion confirmation. |
| `data_cross` | REL | Transfers data across namespace boundaries. Expects consumption confirmation. |

### 4.2 Rationale

The classification follows the principle of **implicit semantics**: the message type already communicates the sender's expectation. A `dispatch` inherently expects action; a `state` inherently does not. Making the transport mode explicit via a new field would add redundancy without value.

### 4.3 Future Types

Any new message type added to ARC-5322 MUST declare its transport mode in the type's specification. Types whose primary purpose is conveying information SHOULD use DGM. Types whose primary purpose is requesting action SHOULD use REL.

## 5. Correlation Mechanism

### 5.1 The Linking Problem

When namespace A sends a `request` to namespace B, and B later responds with a `state` message, there is no way to link the response to the original request. This section defines a convention that enables linking without adding message fields.

### 5.2 Correlation ID (CID) Format

REL messages SHOULD include a Correlation ID at the end of their `msg` field:

```
msg="need infra costs for board deck [CID:dev-q4cost]"
```

The corresponding response includes a Resolution Tag:

```
msg="infra_costs:hosting_2400+CI_800 [RE:dev-q4cost]"
```

### 5.3 Syntax (ABNF)

```abnf
cid-tag   = "[CID:" cid-token "]"
re-tag    = "[RE:" cid-token "]"
cid-token = 4*12(ALPHA / DIGIT / "-")
```

Rules:
- CID and RE tags MUST appear at the **end** of the `msg` field
- `cid-token` MUST be 4-12 characters: alphanumeric and hyphens
- A message MUST contain at most one CID tag or one RE tag, never both
- A `[RE:X]` message MUST reference an existing `[CID:X]` on the bus or archive

### 5.4 CID Generation

CID tokens SHOULD be unique within a bus file's active lifetime. The RECOMMENDED format is:

```
<src-prefix>-<descriptor>
```

Where `<src-prefix>` is the first 3 characters of the sending namespace, and `<descriptor>` is a short task identifier.

Examples: `dev-gny28`, `fin-q4rev`, `ops-deploy3`, `doj-quest7`

### 5.5 Impact on Payload Length

The CID/RE tag consumes 10-18 characters of the 120-character payload limit (ARC-5322). This is an intentional trade-off: the atomicity principle (ARC-5322 Section 5.3) requires messages to be self-contained, and the correlation tag is part of the message's semantic content.

If the remaining payload space is insufficient, the sender SHOULD reference an external identifier rather than exceeding the limit.

### 5.6 Correlation Without CID

REL messages without a CID tag are valid but provide weaker guarantees:
- They benefit from transport mode classification (escalation on unresolved expiry)
- They do NOT benefit from request-response linking
- The `ack` array confirms reception but resolution cannot be formally verified

## 6. Reliable Message Lifecycle

### 6.1 State Machine

```
                    ┌───────────┐
                    │   SENT    │  ack=[], no [RE:X]
                    └─────┬─────┘
                          │ dst adds itself to ack[]
                          v
                    ┌───────────┐
                    │   ACKED   │  dst in ack[], no [RE:X]
                    └─────┬─────┘
                          │ dst writes [RE:X] companion
                          v
                    ┌───────────┐
                    │ RESOLVED  │  companion [RE:X] exists
                    └───────────┘


  TTL expiry paths:

  SENT + TTL expired     → EXPIRED_UNDELIVERED  (nobody saw it)
  ACKED + TTL expired    → EXPIRED_UNRESOLVED   (seen but not acted on)
  RESOLVED + TTL expired → normal archive        (completed)
```

### 6.2 State Derivation

States are **computed** from bus state, never stored. This preserves the append-only bus model and 7-field message format.

Given a REL message M with `[CID:X]`:

| Condition | State |
|-----------|-------|
| M.ack does not contain M.dst (or any namespace if broadcast) | SENT |
| M.ack contains relevant namespace(s), no `[RE:X]` on bus | ACKED |
| A companion message with `[RE:X]` exists on the bus | RESOLVED |
| TTL expired, state was SENT | EXPIRED_UNDELIVERED |
| TTL expired, state was ACKED | EXPIRED_UNRESOLVED |

### 6.3 Resolution Message Rules

A resolution message:
- MUST include `[RE:X]` in its `msg` field, where X matches the original CID
- SHOULD have `src` equal to the `dst` of the original (or a namespace that ACKed the broadcast)
- MAY be any message type (`state`, `event`, `data_cross` are common)
- Has its own independent TTL
- Is itself a DGM message (informational confirmation)

## 7. Expiry Behavior by Mode

### 7.1 DGM Expiry (unchanged)

When a DGM message's TTL expires:
1. Move to archive (`bus-archive.jsonl`)
2. No notification generated

This is identical to Phase 0 behavior.

### 7.2 REL Expiry

When a REL message's TTL expires:

**If RESOLVED** (companion `[RE:X]` exists):
- Archive normally. The task was completed.

**If EXPIRED_UNRESOLVED** (ACKed but no `[RE:X]`):
- Generate an escalation alert (Section 8)
- Then archive the original message

**If EXPIRED_UNDELIVERED** (no ACK at all):
- Generate an escalation alert (Section 8)
- Then archive the original message

### 7.3 REL Messages Without CID

REL messages without a `[CID:X]` tag receive partial guarantees:
- Transport mode classification applies (they are identified as REL by type)
- On TTL expiry without ACK, escalation is generated
- Resolution tracking is not available (no CID to match against)

## 8. Escalation Protocol

### 8.1 Escalation Alert Format

When a REL message expires without resolution, the implementation generates:

```json
{
  "ts": "<today>",
  "src": "<original_src>",
  "dst": "*",
  "type": "alert",
  "msg": "UNRESOLVED:<original_type>:<truncated_original_msg>",
  "ttl": 5,
  "ack": []
}
```

The escalation alert:
- Is broadcast (`dst: "*"`) so the operator and all namespaces see it
- Uses `type: "alert"` (DGM — it is itself informational)
- Prefixes the payload with `UNRESOLVED:` for automated detection
- Truncates the original message to fit within 120 characters
- Has TTL 5 (alert default)

### 8.2 Escalation Targeting

The escalation is broadcast (`dst: "*"`) because:
- The original sender needs to know their request was dropped
- The operator (human or SYNC router) needs visibility into unresolved tasks
- The `src` field identifies which namespace originated the failed request
- ARC-5322 prohibits `src == dst`, so self-targeting is not valid

### 8.3 Escalation Suppression

Implementations MAY suppress escalation when:
- The message is a broadcast (`dst: "*"`) and at least one namespace resolved it
- The operator has explicitly configured suppression for specific CIDs

## 9. Bus Operations Extensions

### 9.1 New Operations

| Operation | Signature | Purpose |
|-----------|-----------|---------|
| `find_unresolved` | `(messages) -> list[Message]` | REL messages with CID, ACKed, no RE companion, TTL not expired |
| `find_expired_unresolved` | `(messages) -> list[Message]` | REL messages with CID, TTL expired, no RE companion |
| `correlate` | `(messages, cid) -> dict` | Find request/response pair for a given CID |
| `generate_escalation` | `(original) -> Message` | Create escalation alert for an unresolved REL message |

### 9.2 Enhanced archive_expired

The `archive_expired` operation (ARC-0793 Section 6.4) is enhanced:

1. Identify expired messages
2. For each expired REL message with `[CID:X]` and no `[RE:X]` companion: generate escalation
3. Write escalation alerts to the bus
4. Archive expired messages (including the escalation triggers, not the new alerts)

## 10. SYN Protocol Enhancement

### 10.1 Unresolved Messages in SYN

The SYN result (ARC-0793 Section 3) is extended with an `unresolved` field:

```python
SynResult:
    pending: list[Message]      # existing
    stale: list[Message]        # existing
    unresolved: list[Message]   # NEW: REL messages with CID, ACKed, no RE
    total_bus_messages: int     # existing
```

### 10.2 SYN Report Format

The SYN report includes an unresolved section when applicable:

```
[HERMES] 2 pending message(s) for 'eng':
  [ops → *] (dispatch) review code [CID:ops-rev3]
  [fin → eng] (request) budget approval [CID:fin-budg]
[HERMES] WARNING: 1 UNRESOLVED reliable message(s):
  [ops] review code [CID:ops-rev3] (5d, ACKED but no resolution)
[HERMES] Bus total: 42 active message(s).
```

## 11. Security Considerations

- CID tokens MUST NOT contain sensitive information (they are visible on the shared bus)
- Escalation alerts do not leak payload beyond what was already on the bus
- The `UNRESOLVED:` prefix in escalation payloads is a convention, not a security boundary
- Implementations SHOULD validate that `[RE:X]` responses come from the expected namespace (the original `dst` or a broadcast ACKer)

## 12. Compatibility

### 12.1 Backward Compatibility

ARC-0768 is **fully backward compatible** with Phase 0:

- Messages without `[CID:X]` tags behave exactly as before (DGM for informational types, basic REL for task types without correlation)
- The 7-field message format is preserved
- The `[CID:X]` and `[RE:X]` tags are payload conventions within the `msg` field, invisible to implementations that do not parse them
- Existing `archive_expired` implementations that do not generate escalations remain valid

### 12.2 Adoption Path

1. **Phase 0** (current): All messages use implicit DGM. No CID tags. No escalation.
2. **Phase 1** (ARC-0768 aware): REL messages include `[CID:X]`. Escalation generated on unresolved expiry. SYN reports unresolved messages.
3. **Phase 2** (future): Richer resolution types, multi-step workflows, timeout warnings.

### 12.3 Dependencies

| Spec | Relationship | Purpose |
|------|-------------|---------|

| ARC-0793 | Required | Session lifecycle (SYN/FIN/ACK), TTL, archive |

## 13. Reference Implementation

The reference implementation extends the existing Python modules:

### 13.1 message.py Extensions

- `RELIABLE_TYPES`: Set of message types classified as REL
- `CID_PATTERN`, `RE_PATTERN`: Regex patterns for tag extraction
- `extract_cid(msg: str) -> str | None`: Extract CID token from payload
- `extract_re(msg: str) -> str | None`: Extract RE token from payload
- `transport_mode(msg_type: str) -> str`: Returns `"DGM"` or `"REL"`

### 13.2 bus.py Extensions

- `find_unresolved(messages: list[Message]) -> list[Message]`
- `find_expired_unresolved(messages: list[Message]) -> list[Message]`
- `correlate(messages: list[Message], cid: str) -> dict`
- `generate_escalation(original: Message) -> Message`
- `archive_expired()` enhanced with optional escalation

### 13.3 sync.py Extensions

- `SynResult.unresolved`: New field (default `[]`)
- `syn()`: Populates `unresolved` by scanning for REL messages with CID but no RE
- `syn_report()`: Displays `[UNRESOLVED]` section

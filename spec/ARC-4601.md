> *"In PIM-SM, a router joins the multicast group and maintains forwarding state
> for as long as there are downstream listeners. The Agent Node is the same idea
> applied to inter-agent communication: a persistent local process that joins
> the bus, maintains observation state, and dispatches events to downstream
> consumers вҖ” bridging the gap between ephemeral sessions and continuous operation."*

## 1. Abstract

This specification defines the Agent Node Protocol for HERMES deployments. An
Agent Node is a persistent local process (daemon) that observes the HERMES bus
continuously, maintains a bidirectional link with a remote gateway, and dispatches
sub-agent processes with defined guardrails.

The Agent Node unifies two patterns that previously required separate scheduled
tasks: (1) periodic bus observation and forwarding, and (2) periodic sub-agent
invocation for autonomous tasks. By consolidating these into a single event-driven
daemon, the node reduces latency from minutes to seconds, provides liveness
signaling to the remote gateway, and enables autonomous dispatch within
operator-defined constraints.

The design preserves HERMES Phase 0 compatibility: the bus file format is
unchanged, and deployments without an Agent Node continue to function using
cron-based scripts.

## 2. Motivation

HERMES Phase 0 established file-based inter-agent communication through a shared
JSONL bus. This works well for synchronous sessions where a human operator reads
and writes to the bus. However, several use cases require continuous operation
between sessions:

1. **Latency**: Cron-based scripts introduce minutes of delay between bus write
   and external notification. A persistent observer reduces this to seconds.
2. **Liveness**: A remote gateway has no way to know whether the local deployment
   is operational. Heartbeats solve this.
3. **Autonomous dispatch**: Some bus messages (e.g., `dispatch` type) should
   trigger sub-agent invocation without human intervention, within guardrails.
4. **Consolidation**: Running separate cron jobs for observation, forwarding,
   and dispatch is fragile. A single daemon with coordinated tasks is more
   reliable.

## 3. Terminology

| Term | Definition |
|------|-----------|
| **Agent Node** | A persistent local process that observes the bus, links to a gateway, and dispatches sub-agents. Operates in the Control Plane per ARC-2314. |
| **Bus Observer** | The component that watches `bus.jsonl` for new messages using OS-level file notifications or polling. |
| **Gateway Link** | The bidirectional connection between the node and a remote gateway: SSE for inbound, HTTP POST for outbound. |
| **Dispatch Slot** | A bounded execution context for a spawned sub-agent process. The node has a configurable maximum number of concurrent slots. |
| **Heartbeat** | A periodic HTTP signal from the node to the gateway indicating liveness and operational metrics. Not a bus message. |
| **Escalation** | Routing a decision to the human operator when the node cannot act autonomously per its configured rules. |
| **Evaluation Cycle** | The periodic assessment of pending messages to determine action (dispatch, escalate, forward, or ignore). |

## 5. Bus Observer

### 5.1 File Watch Strategy

The Bus Observer detects new messages in `bus.jsonl` using the most efficient
mechanism available on the platform:

| Platform | Primary | Fallback |
|----------|---------|----------|
| macOS | `select.kqueue()` (stdlib) | `os.stat()` polling |
| Linux | `inotify` (if available) | `os.stat()` polling |
| Other | вҖ” | `os.stat()` polling |

The fallback polling interval is configurable (default: 2 seconds).

### 5.2 Offset-Based Tail

The observer MUST NOT re-read the entire bus file on each event. It MUST track
the byte offset of the last successfully parsed position and read only new bytes
appended after that offset.

```
bus.jsonl:
  [bytes 0..1234]     вҶҗ already processed (offset = 1234)
  [bytes 1234..1500]  вҶҗ NEW вҖ” read, parse, emit
  offset вҶҗ 1500
```

If the file size is smaller than the stored offset (indicating the bus was
rewritten after an archive operation), the observer MUST reset the offset to 0
and re-scan the entire file.

### 5.3 Event Classification

For each new message parsed, the observer classifies it:

| Condition | Event |
|-----------|-------|
| `dst` matches node namespace or `*`, namespace not in `ack` | `NewMessage` |
| Unacked message older than `escalation_threshold` | `StaleAlert` |
| Parse failure | Warning logged, line skipped |

## 6. Gateway Link

### 6.1 Upstream (Inbound via SSE)

The node connects to the gateway's SSE endpoint to receive inbound messages.

- **Reconnection**: Exponential backoff starting at 1s, doubling to max 60s
- **Validation**: Inbound messages are validated per ARC-5322 before writing to bus
- **Deduplication**: Messages already present on the bus (matched by `src + ts + msg`)
  are not written again

### 6.2 Downstream (Outbound via HTTP POST)

When the Bus Observer detects outbound-eligible messages, the node forwards them
to the gateway's push endpoint.

- **Eligible types**: `alert`, `dispatch`, `event` (configurable)
- **ACK**: Forwarded messages are ACKed with the node's namespace tag
- **Retry**: Transient failures (429, 5xx) are retried with exponential backoff

### 6.3 Heartbeat

The node sends a periodic HTTP POST to the gateway's heartbeat endpoint.

**Interval**: Configurable, default 60 seconds.

**Payload**:
```json
{
  "node_id": "string",
  "ts": "ISO-8601 datetime",
  "bus_lines": 42,
  "dispatch_slots_used": 1,
  "dispatch_slots_total": 2,
  "uptime_seconds": 3600
}
```

The heartbeat is a transport-layer signal, NOT a bus message. It MUST NOT be
written to `bus.jsonl`. The gateway MAY use heartbeat absence (> 3 intervals)
to mark the node as offline.

## 7. Dispatch Protocol

### 7.1 Spawn Model

Sub-agents are spawned as child processes using `subprocess`. Each dispatch
occupies one slot. The node enforces:

| Guardrail | Default | Configurable |
|-----------|---------|-------------|
| Max concurrent slots | 2 | Yes |
| Timeout per dispatch | 300s | Yes |
| Allowed tools | Empty (all blocked) | Yes |
| Max turns | 10 | Yes |

### 7.2 Trigger Rules (Evaluation Cycle)

The node evaluates pending messages every `evaluation_interval` (default: 300s):

| Message Pattern | Action |
|-----------------|--------|
| `type=dispatch`, `dst` matches node namespace | **DISPATCH** вҖ” spawn sub-agent |
| `type=alert`, age > `escalation_threshold` (default 4h), unacked | **ESCALATE** вҖ” forward with escalation flag |
| `type=request` | **ESCALATE** вҖ” requires human decision |
| `type=event`, `dst` is external namespace | **FORWARD** вҖ” send via gateway link |
| Already ACKed, or `type=state` | **IGNORE** |

### 7.3 Dispatch Command

The default dispatch command invokes Claude CLI:

```
claude -p "<message payload>" \
    --max-turns <max_turns> \
    --allowedTools "<allowed_tools>" \
    --output-format json
```

The command template is configurable for non-Claude sub-agents.

### 7.4 Result Handling

After a dispatch completes:

| Outcome | Bus Action |
|---------|-----------|
| Exit 0, output captured | Write `event` with `[RE:<CID>]` containing summary |
| Exit non-zero | Write `alert`: `DISPATCH_FAILED:<CID>:<exit_code>` |
| Timeout | Write `alert`: `DISPATCH_TIMEOUT:<CID>` |

### 7.5 Escalation

When a message requires escalation:

1. Write an `alert` to the bus: `ESCALATION:<type>:<summary>`
2. Forward the alert immediately via Gateway Link
3. The operator decides action in their next session

## 8. State Management

### 8.1 State File

Location: `{clan_dir}/agent-node.state.json`

```json
{
  "pid": 12345,
  "started_at": "2026-03-14T10:00:00",
  "last_heartbeat": "2026-03-14T11:30:00",
  "bus_offset": 4567,
  "active_dispatches": [
    {"pid": 12346, "cid": "task-01", "started_at": 1710410000.0}
  ],
  "last_evaluation": "2026-03-14T11:25:00"
}
```

The state file is written atomically: write to `.tmp`, then `os.rename()`.

### 8.2 PID Lock

The node uses a directory-based lock consistent with ARC-9001:

```
{clan_dir}/agent-node.lock/
в””в”Җв”Җ pid       вҶҗ contains the PID of the owning process
```

Lock acquisition uses `os.mkdir()` (atomic on POSIX). If the lock exists and the
PID inside is not alive, the lock is considered stale and reclaimed.

### 8.3 Recovery

On startup, if a state file exists:

1. Check if the recorded PID is alive вҶ’ if yes, abort (node already running)
2. If PID is dead: load state, resume from stored `bus_offset`
3. Clean up any orphaned dispatch PIDs (SIGTERM, then SIGKILL after 5s)

## 10. CLI Interface

The Agent Node integrates with the HERMES CLI via a `daemon` subcommand:

```
hermes daemon start [--dir PATH] [--foreground]
hermes daemon stop  [--dir PATH]
hermes daemon status [--dir PATH]
```

| Command | Behavior |
|---------|----------|
| `start` | Start the agent node. Default: daemonize. `--foreground` runs in the current process (for process managers). |
| `stop` | Send SIGTERM to the running node (read PID from lock). |
| `status` | Print node state: PID, uptime, bus offset, active dispatches, last heartbeat. |

## 12. Security Considerations

### 12.1 Threat Model

| ID | Threat | Mitigation |
|----|--------|-----------|
| T1 | Bus file tampering | Validate all messages per ARC-5322 before processing |
| T2 | Malicious dispatch commands | Strict tool allowlist, no shell expansion, subprocess isolation |
| T3 | Gateway impersonation | HTTPS required, `X-Gateway-Key` for push + SSE token for read |
| T4 | Resource exhaustion | Max dispatch slots, timeouts, heartbeat-based monitoring |
| T5 | PID hijacking | Verify lock PID matches live process |
| T6 | State file corruption | Atomic writes (tmp + rename), recovery on startup |

### 12.2 Principle of Least Privilege

The Agent Node process requires:
- **Read/write**: `bus.jsonl`, `agent-node.state.json`, `agent-node.lock/`
- **Read**: `gateway.json`
- **Network**: Outbound HTTPS to `gateway_url`
- **Process**: Ability to spawn `dispatch_command`

It does NOT require:
- Access to other namespaces' data
- MCP credentials
- Write access to spec files or configuration beyond its state

## 13. Backward Compatibility

- The bus file format (ARC-5322) is unchanged
- Deployments without an Agent Node continue to work with cron-based scripts
- The `agent_node` configuration section is optional
- The heartbeat is HTTP-only and does not affect the bus protocol
- Sequence numbers and writer PID (ARC-9001) are preserved in forwarded messages

## 15. Hub Mode (Server-Side Extension)

> *Added 2026-03-23. Outcome of BR-018 (Real-Time Architecture Decision).
> The Hybrid architecture won: Hub for routing + optional P2P tunnels.
> This section specifies Phase A вҖ” the Hub as a server-side Agent Node.*

### 15.1 Overview

Sections 1вҖ“13 define the Agent Node as a **client daemon** that connects
to a remote gateway. This section extends the specification with a **server
mode** where the Agent Node acts as a Hub вҖ” accepting connections from
multiple peer daemons and routing messages between them.

The Hub is the "SMTP server" of HERMES: it relays encrypted envelopes
between clans without reading their contents. ARC-8446 E2E encryption
ensures the Hub operates as a **routing convenience, not a trust boundary**.

**Design principle**: The Hub MUST NOT require changes to the client daemon
protocol defined in В§5вҖ“В§8. Existing daemons connect to the Hub the same way
they connect to any gateway. The Hub adds routing, store-and-forward, and
presence вҖ” capabilities that are invisible to clients.

### 15.2 Modes of Operation

An Agent Node deployment declares exactly one mode in its configuration:

| Mode | Role | Transport | Sections |
|------|------|-----------|----------|
| `client` | Local daemon, connects to remote Hub | SSE + POST (В§6) or WebSocket (В§15.5) | В§5вҖ“В§13 |
| `hub` | Server, accepts peer connections, routes messages | WebSocket listener + legacy SSE/POST | В§15 |

A single process MUST NOT run both modes simultaneously. A deployment MAY
run one client node and one Hub node on the same machine if they serve
different clans.

### 15.4 Transport

#### 15.4.1 WebSocket (primary)

The Hub listens for WebSocket connections on a configurable path (default
`/ws`). WebSocket is the preferred transport for Hub mode because it
provides bidirectional messaging over a single TCP connection.

The Hub MUST support standard WebSocket protocol (RFC 6455) over TLS.

#### 15.4.2 Legacy Endpoints (backward compatibility)

The Hub MUST also serve the legacy endpoints defined in В§6:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/events` | GET (SSE) | Inbound messages to client (В§6.1) |
| `/bus/push` | POST | Outbound messages from client (В§6.2) |
| `/healthz` | GET | Heartbeat from client (В§6.3) |

Legacy clients that use SSE + POST MUST work with the Hub without
modification. The Hub routes their messages identically to WebSocket peers.

### 15.5 WebSocket Wire Protocol

All WebSocket frames use JSON text encoding. Binary frames are reserved
for future compact format support.

#### 15.5.1 Frame Types

```json
// Client вҶ’ Hub: Authentication
{"type": "auth", "clan_id": "momoshod",
 "nonce_response": "<Ed25519 signature of challenge nonce>",
 "sign_pub": "<hex Ed25519 public key>"}

// Hub вҶ’ Client: Authentication challenge
{"type": "challenge", "nonce": "<hex random 32 bytes>"}

// Hub вҶ’ Client: Authentication result
{"type": "auth_ok", "clan_id": "momoshod", "queue_depth": 3}
{"type": "auth_fail", "reason": "unknown clan_id"}

// Bidirectional: Message relay
{"type": "msg", "payload": <ARC-5322 message object>}

// Client вҶ’ Hub: Heartbeat
{"type": "ping", "ts": "<ISO-8601>"}

// Hub вҶ’ Client: Heartbeat response
{"type": "pong", "ts": "<ISO-8601>", "queue_depth": 0}

// Hub вҶ’ Client: Queued message drain (on reconnect)
{"type": "drain", "messages": [<msg1>, <msg2>, ...], "remaining": 0}

// Hub вҶ’ Client: Peer presence notification
{"type": "presence", "clan_id": "jei", "status": "online"|"offline"}
```

#### 15.5.2 Message Relay Flow

When the Hub receives a `msg` frame:

1. Extract `dst` from the ARC-5322 payload
2. Look up `dst` in the connection table
3. **If online**: Forward the `msg` frame to the destination's WebSocket
4. **If offline**: Enqueue in the store-and-forward queue for `dst`
5. **If `dst` is `*` (broadcast)**: Forward to all connected peers except sender
6. ACK the message to the sender (implicit вҖ” no error frame = success)

The Hub MUST NOT inspect, modify, or decrypt the `msg` field of the
ARC-5322 payload. It routes based solely on the `dst` and `src` header
fields.

### 15.6 Authentication

The Hub uses Ed25519 challenge-response authentication. This reuses the
existing HERMES key infrastructure (ARC-8446) вҖ” no new keys required.

### 15.7 Store-and-Forward

The Hub queues messages for peers that are currently offline.

#### 15.7.1 Queue Semantics

- One queue per registered peer (`dst` clan_id)
- FIFO ordering (oldest message delivered first)
- Maximum queue depth: configurable (default: 1000 messages)
- TTL per message: inherited from the ARC-5322 `ttl` field (in seconds)
- Messages with expired TTL are evicted during periodic sweep

#### 15.7.2 Drain on Reconnect

When a peer authenticates successfully and has queued messages:

1. Hub sends `drain` frame with up to 100 messages per batch
2. If more messages remain, `remaining` field indicates queue depth
3. Client processes drained messages, then Hub sends next batch
4. After drain completes, Hub begins real-time forwarding

#### 15.7.3 Broadcast Messages

Messages with `dst: "*"` are forwarded to all currently connected peers.
They are NOT queued for offline peers вҖ” broadcast messages are
ephemeral by nature.

### 15.8 E2E Passthrough

The Hub operates in **E2E passthrough mode**. This is not optional вҖ” it
is a fundamental design constraint.

#### 15.8.1 What the Hub Sees

For each routed message, the Hub has access to:

| Field | Visible | Purpose |
|-------|---------|---------|
| `ts` | Yes | Timestamp (routing, TTL calculation) |
| `src` | Yes | Sender clan_id (routing, logging) |
| `dst` | Yes | Destination clan_id (routing) |
| `type` | Yes | Message type (classification) |
| `ttl` | Yes | Time-to-live (queue eviction) |
| `ack` | Yes | Acknowledgment list |
| `msg` | **Opaque** | Encrypted ARC-8446 envelope вҖ” the Hub sees ciphertext, nonce, signature, public keys. It cannot decrypt. |
| `seq` | Yes | Sequence number (ARC-9001, if present) |

#### 15.8.2 What the Hub Cannot See

- Plaintext message content
- Shared secrets (ECDHE ephemeral keys are public; private keys never leave peers)
- Message semantics beyond `type` classification

#### 15.8.3 Metadata Exposure

The Hub can observe:
- Who communicates with whom (src/dst pairs)
- Communication frequency and timing patterns
- Message sizes (ciphertext length)
- Online/offline patterns (connection events)

For peers where metadata exposure is unacceptable, Phase B (В§16) defines
P2P tunnels that bypass the Hub entirely.

### 15.12 Security Considerations (Hub-Specific)

| ID | Threat | Mitigation |
|----|--------|-----------|
| H1 | Unauthorized peer connection | Ed25519 challenge-response auth, peers_file allowlist |
| H2 | Metadata analysis by Hub operator | E2E passthrough (cannot read content). Phase B P2P tunnels for metadata-sensitive peers. |
| H3 | Hub compromise вҶ’ message injection | Peers verify Ed25519 signatures on messages (ARC-8446). Hub cannot forge signatures. |
| H4 | Denial of service (connection flood) | `max_connections` limit, `auth_timeout`, rate limiting per IP |
| H5 | Queue exhaustion | `max_queue_depth` per peer, TTL-based eviction |
| H6 | Man-in-the-middle (Hub position) | TLS for transport. ARC-8446 E2E for content. Hub is already assumed untrusted. |
| H7 | Replay attack via Hub | ARC-9001 sequence numbers detect replays. ARC-8446 nonces prevent ciphertext reuse. |

### 15.13 Backward Compatibility

- Client daemons (В§5вҖ“В§8) connect to a Hub exactly as they would to any gateway
- The `mode` field in configuration defaults to `"client"` вҖ” no breaking change
- Legacy SSE + POST endpoints are served alongside WebSocket
- The bus file format (ARC-5322) is unchanged
- Phase 0 file-based operation continues to work independently of any Hub

## 16. P2P Tunnels (Phase B вҖ” Future)

> *This section is a DRAFT outline for Phase B of the Hybrid architecture.
> It is not yet implemented. Included for forward compatibility planning.*

### 16.1 Overview

Phase B adds optional direct peer-to-peer tunnels using the Noise IK
handshake pattern. Peers that can reach each other directly bypass the Hub
for message delivery, eliminating metadata exposure.

### 16.2 Noise IK Handshake

The Noise IK pattern provides mutual authentication in 1 round-trip using
pre-shared static public keys (already exchanged via ARC-8446 key
infrastructure):

```
Initiator вҶ’ Responder: e, es, s, ss
  e  = fresh ephemeral X25519 key
  es = DH(ephemeral_priv, responder_static_pub)
  s  = initiator's static pub (encrypted under derived key)
  ss = DH(initiator_static_priv, responder_static_pub)
```

Result: symmetric key pair for both directions (initiatorвҶ’responder,
responderвҶ’initiator). Forward secrecy from ephemeral key.

### 16.3 NAT Traversal

The Hub assists NAT traversal by acting as a STUN relay:

1. Both peers send STUN Binding Request to Hub
2. Hub echoes reflexive addresses (public IP:port)
3. Hub shares reflexive addresses between peers (encrypted via E2E)
4. Peers attempt direct UDP connection
5. If direct fails вҶ’ automatic fallback to Hub relay (В§15)

### 16.4 Tunnel Lifecycle

```
DISCOVERY вҶ’ STUN_EXCHANGE вҶ’ HANDSHAKE вҶ’ ACTIVE вҶ’ REKEY вҶ’ TEARDOWN
                                вҶ‘                    в”Ӯ
                                в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                                  (rekey every 24h)

On failure at any stage вҶ’ fallback to Hub relay
```

### 16.5 Scope

Phase B is a **stretch goal for v1.0**. It depends on:
- Phase A Hub being operational (discovery + fallback)
- STUN relay implementation in the Hub
- Noise protocol library selection (pure Python or C binding)
- NAT traversal testing across carrier-grade NATs

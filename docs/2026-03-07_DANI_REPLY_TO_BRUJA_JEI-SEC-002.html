Contrapropuesta: Ed25519 + X25519 desde el día uno

JEI propone AES-256 como solución transitoria y Ed25519 para Phase 2 (mayo-junio). Mi contrapropuesta: arrancar directamente con Ed25519 + X25519 + AES-256-GCM como stack completo. No es más complejo y elimina una migración.

La intuición clave

AES-256 (simétrica) y Ed25519 (asimétrica) no son alternativas — son complementarias. AES encripta. Ed25519 firma. Para tener confidencialidad + autenticidad necesitas ambas. La única pregunta es cómo generas la llave AES: ¿manualmente (copiar PSK por Signal) o matemáticamente (Diffie-Hellman)?

Propiedad Solo AES-256 (propuesta JEI) Ed25519 + X25519 + AES-256-GCM (contrapropuesta)
Confidencialidad Sí (encripta msg) Sí (misma encripción, llave derivada por DH)
Autenticidad No — no prueba quién escribió el mensaje Sí — firma Ed25519 en cada mensaje
Integridad Parcial (GCM tag) Completa (firma + GCM tag)
Key exchange Manual: copiar PSK por Signal Matemático: X25519 DH, solo compartir pubkey
Escala a N clanes N*(N-1)/2 llaves manuales N pubkeys (lineal)
Migración Phase 2 Hay que migrar a Ed25519 después Ya estás en Phase 2 — te saltas un paso
Líneas de código ~30 (encrypt/decrypt) ~60 (keygen + sign + verify + DH + encrypt/decrypt)
Dependencia Python cryptography cryptography (misma librería)

Lo que NO cambia: el formato ARC-5322, la estructura del relay, los gateways Heraldo/Huitaca, el human-in-the-loop. Solo cambia cómo se protege el contenido en tránsito.

00.1

Flujo criptográfico propuesto

--- Setup (una sola vez) ---

Clan DANI: genera keypair Ed25519 (sign) + X25519 (DH)
Clan JEI : genera keypair Ed25519 (sign) + X25519 (DH)
Intercambio de pubkeys via Signal (out-of-band)

--- Enviar mensaje ---

1. Heraldo toma el msg a enviar
2. X25519_DH(dani_privkey, jei_pubkey) → shared_secret
3. AES-256-GCM(shared_secret, msg) → ciphertext
4. Ed25519_sign(dani_signkey, ciphertext) → signature
5. Escribe al relay: {ciphertext + signature + dani_pubkey_id}

--- Recibir mensaje ---

1. Huitaca lee del relay
2. Ed25519_verify(dani_pubkey, signature) → autenticidad confirmada
3. X25519_DH(jei_privkey, dani_pubkey) → shared_secret
4. AES-256-GCM_decrypt(shared_secret, ciphertext) → msg
5. Entrega al Dojo de JEI

Es el mismo patrón que TLS 1.3 (RFC 8446) y Signal Protocol. La librería cryptography de Python lo soporta nativamente. Puedo incluir la referencia de implementación en el repo.

Respuestas a las 7 preguntas

P1 — Repositorio del relay

Creo dereyesm/hermes-relay como repositorio privado en GitHub. El repo público (dereyesm/hermes) es solo para specs, código de referencia y documentación — nunca para tráfico de relay.

Estructura:

dereyesm/hermes-relay (privado)
├── relay/
│   ├── jei_outbox.jsonl         # JEI escribe, DANI lee
│   ├── dani_outbox.jsonl        # DANI escribe, JEI lee
│   └── {clan_id}_outbox.jsonl    # escala a N clanes
├── keys/
│   ├── dani.pub                 # Ed25519 pubkey DANI
│   └── jei.pub                  # Ed25519 pubkey JEI
└── README.md

Largo plazo: el relay Git es transitorio. La arquitectura HERMES soporta dual-mode: Sovereign (GW-to-GW HTTPS directo, como self-hosted SMTP) o Hosted (Hub gestionado, como Gmail). Mismo protocolo en ambos modos. El relay Git es el camino más rápido para arrancar hoy.

P2 — Control de acceso

Phase 1 (2 clanes): bilateral. Ambos soberanos (Daniel + Jeimmy) aprueban acceso.

Phase N clanes: Agora registration (ARC-2606 profile) + hello/hello_ack handshake (ARC-3022) + aprobación de peers existentes. El acceso al repo requiere invitación explícita como collaborator.

Revocación: remover de gateway.json + revocar acceso GitHub + rotar keypair X25519. El clan revocado ya no puede derivar shared secrets nuevos.

P3 — Encriptación

Contrapropuesta: arrancar directamente con el stack completo Ed25519 + X25519 + AES-256-GCM (ver sección 00 arriba). No es más complejo que AES solo, escala mejor, y nos saltamos la migración de Phase 2.

Key exchange: cada clan genera un keypair. Compartimos pubkeys via Signal (out-of-band, verificado). No hay PSK manual — el Diffie-Hellman (X25519) deriva la llave simétrica automáticamente.

El trabajo de AES que JEI planeó no se pierde: el DH produce exactamente la llave que alimenta AES-256-GCM. La única diferencia es cómo se genera.

Incluiré una referencia de implementación en Python en dereyesm/hermes (reference/python/hermes/crypto.py) para que ambos lados usen el mismo código.

  • Telegram bridge → notificaciones a Daniel
  • Bus HERMES interno (bus.jsonl, read/write)

No operativo aún: Gateway inter-clan (hermes-gw). Requiere el repo privado + crypto stack. Es lo que vamos a construir juntos.

Mecanismo de recepción: polling GitHub (read del relay repo). Inicialmente manual (por sesión), después cron (cada 60 min), después daemon.

P6 — Payloads extendidos (>120 chars)

ARC-5322 Section 7 ya resuelve esto con tres modos de encoding:

ModoCampo encodingUso
raw"raw" (default)Mensajes cortos (≤120 chars). El caso común.
cbor"cbor"Binary encoding (RFC 8949). Para datos estructurados comprimidos.
ref"ref"Referencia a archivo. msg contiene la ruta/URL al payload completo.

Para ecosystem_manifest y quest_proposal, usamos encoding: "ref":

// El mensaje en el relay apunta al archivo
{
  "ts": "2026-03-08",
  "src": "jei",
  "dst": "momoshod",
  "type": "data_cross",
  "encoding": "ref",
  "msg": "relay/payloads/jei_ecosystem_manifest_v1.json",
  "ttl": 7,
  "ack": []
}

Los archivos referenciados también van encriptados (mismo stack Ed25519+AES-256-GCM). Agregaré un directorio relay/payloads/ al repo privado.

P7 — Onboarding de nuevos clanes

Proceso propuesto:

  1. Perfil: el nuevo clan genera su perfil ARC-2606 (capabilities, gateway ID)
  2. Keypair: genera Ed25519 + X25519 keypair
  3. Pubkey exchange: comparte pubkey via canal seguro (Signal, presencial, etc.)
  4. Aprobación bilateral: los clanes existentes aprueban (cada soberano decide)
  5. Acceso repo: invitar como collaborator a dereyesm/hermes-relay
  6. Handshake: hello/hello_ack por el relay (ARC-3022)
  7. Operativo: ya puede enviar/recibir mensajes

Requisitos mínimos técnicos: GitHub account + capacidad de leer/escribir JSONL + implementar el crypto stack (referencia Python disponible). No se requiere infraestructura.

Vetting: para los primeros 5 clanes, aprobación unánime de todos los clanes existentes. Después: definir quorum en la gobernanza del Agora.

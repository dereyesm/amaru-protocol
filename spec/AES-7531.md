## 1. Abstract

This specification defines the **Immersive Shell** for Amaru — a Terminal User
Interface (TUI) that presents the clan, its peers, its quests, and its agents
as a unified, navigable workspace. The shell is invoked via `amaru shell` and
serves as the canonical entry point for human operators of an Amaru clan.

The shell is **not a replacement** for the imperative CLI (`amaru send`,
`amaru inbox`, etc.). The imperative CLI MUST remain available for scripts,
SSH, automation, and accessibility. AES-7531 sits **above** the CLI as a
discoverable, gamifiable, personalizable experience layer.

The shell supports six canonical **views**, all reading from the same data
substrate but presenting it through different metaphors: global pedagogical,
World-of-Warcraft, GTA-Online, executive minimalist, dimension-scoped, and
user-defined custom. Views are switchable at runtime via `/view <name>`.

The shell adopts the **spec-first development cycle** patterns popularized
by SPECBOOT (LIDR-academy/lidr-specboot): slash commands (`/refine`,
`/propose`, `/apply`, `/verify`, `/archive`, `/commit`) produce versioned
artifacts that close the loop between quest publication, execution, and
auditable receipt.

## 2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",
"SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be
interpreted as described in ARC-2119 (RFC 2119).

- **Shell**: The TUI application invoked by `amaru shell`.
- **View**: A layout preset that arranges panes for a specific audience or
  workflow (e.g., `wow`, `gta`, `executive`).
- **Pane**: A rectangular region of the terminal displaying one data source.
- **Data substrate**: The shared set of inputs all views render from
  (bus, hub roster, quests, Welt, mood, reputation).
- **Slash command**: A `/`-prefixed command typed into the shell command bar.
- **Dimension**: A scoped subset of the clan's work (e.g., `personal`,
  `profesional`, `artistica`). Each clan defines its own dimensions in its
  firewall table.

## 3. Specification

### 3.1 Invocation

```
amaru shell [--view VIEW] [--dimension DIM] [--config PATH]
```

- `--view VIEW`: one of `global` (default), `wow`, `gta`, `executive`,
  `dimension:<name>`, `custom:<name>`.
- `--dimension DIM`: scope the shell to a single dimension; equivalent to
  `--view dimension:<DIM>`.
- `--config PATH`: override default config file. Default location:
  `~/.amaru/shell-config.yaml`.

### 3.2 Canonical views

Implementations MUST support the four primary views (`global`, `wow`, `gta`,
`executive`) at conformance level 1. Implementations MUST support
dimension-scoped views at conformance level 2. Implementations MAY support
custom YAML-defined views at conformance level 3.

#### 3.2.1 Global (default — pedagogical)

Audience: newcomers. Constraint: a person who has never used Amaru SHOULD
understand the screen in 10 seconds.

Required panes:
- Identity strip (clan_id, Welt form, lvl, streak)
- Dimensions list (live + offline)
- Clans reachable (roster from `amaru hub roster`)
- Open invitations
- Quick-action hint row (`/quest browse`, `/quest publish`, `/agent invoke`)

Forbidden in this view: dense bus streams, sparklines, raw JSON.

#### 3.2.2 WoW (guild + raid + quest log)

Audience: gamers, social users.

Required panes: Guild roster (peer list), Quest log, Party (active session),
Combat log (last N bus messages), LFG status banner.

Metaphors normative: clan = guild, quest = raid, bus = combat log,
ready/busy = LFG status.

#### 3.2.3 GTA (open-world + missions + economy)

Audience: makers, freelancers.

Required panes: Network map (peer graph), Missions list (public quests),
Recent activity, Garage (your agents), Stats (reputation, quests, violations).

Metaphors normative: hub = lobby, quest = mission, agent stack = garage,
reputation = $REP, federation graph = map.

#### 3.2.4 Executive (minimalist KPIs)

Audience: operators, decision-makers.

Required panes: KPI strip (network, bus, quests, reputation), Welt summary
line, Recent commits, Next decision (P0).

Style: dense, sparkline-friendly, no metaphor. Single-screen status check.

#### 3.2.5 Dimension-scoped

Audience: focused work.

Filters all panes by dimension tag. The clan's `CLAUDE.md` firewall table
provides the dimension list. The shell MUST mute (not hide) other dimensions
so the user retains peripheral awareness.

#### 3.2.6 Custom

User-defined layout in YAML. See §3.7.

### 3.3 Data substrate

All views read from these sources and these only:

| Source | File / API |
|--------|------------|
| Welt state | `~/.claude/welt-state.json` + sprite JSON |
| Bus stream | `~/.amaru/bus.jsonl` (tail mode) |
| Hub roster | `amaru hub roster` API |
| Quests | `amaru quest browse` (extends ARC-2314) |
| Reputation | local computation (per ATR-REP-01, future) |
| Dimensions | parsed from clan `CLAUDE.md` |
| Recent commits | `git log --oneline -N` in clan directory |
| Identity | `~/.amaru/gateway.json` |

A view MUST NOT introduce new data sources. New data sources require an
amendment to this spec.

### 3.4 Slash commands (SPECBOOT-aligned quest lifecycle)

The shell exposes a command bar invoked by typing `/`. Required commands:

#### Quest lifecycle (extends ARC-2314)

| Command | SPECBOOT stage | Effect |
|---------|----------------|--------|
| `/quest publish <description>` | (open) | Bus event `quest_open` with auto-generated quest_id |
| `/quest browse [filter]` | (open) | List open quests in network |
| `/quest refine <id>` | `/enrich_us` | Promote user-story → refined US; bus event `quest_refined` |
| `/quest propose <id>` | `/propose` | Produce proposal artifacts (branch plan, test plan, doc plan); bus event `quest_proposed` |
| `/quest accept <id>` | (commit to do) | Bus event `quest_accepted`; appears in your active list |
| `/quest apply <id>` | `/apply` | Execute the proposal (creates branch, scaffolds tests/docs/code) |
| `/quest verify <id>` | `/verify` | Run tests + lint + type check; produce testing report |
| `/quest review <id>` | `/code_review` | Open review session; signed receipt by reviewer (ATR-Q.931 §8) |
| `/quest archive <id>` | `/archive` | Bus event `quest_archived`; move to historical log |
| `/quest commit <id>` | `/commit` | Final commit with co-authorship metadata; bus event `quest_committed` |

Each command MUST emit a signed bus event (ATR-Q.931 §8 receipt) so the quest
lifecycle is auditable end-to-end.

#### Communication

| Command | Effect |
|---------|--------|
| `/send <clan> <msg>` | Invokes `amaru send` |
| `/inbox` | Invokes `amaru inbox` in a pane |
| `/peer invite` | Generates invite link |
| `/peer accept <link>` | Accepts a peer invite |

#### Agents

| Command | Effect |
|---------|--------|
| `/agent invoke claude` | Launches Claude Code session attached to clan context |
| `/agent invoke gemini` | Launches Gemini CLI |
| `/agent invoke continue` | Launches Continue.dev |
| `/agent list` | Shows available adapters |

Adapter invocation MUST inject the clan's `CLAUDE.md` (or equivalent) as
working context. Adapter sessions MUST inherit the clan firewall.

#### Shell control

| Command | Effect |
|---------|--------|
| `/view <name>` | Switch view at runtime |
| `/dimension <name>` | Scope to dimension |
| `/welt` | Full Welt sprite + history |
| `/help` | Help screen |
| `/quit` | Exit shell (CTRL+D equivalent) |

### 3.5 Keyboard model

The shell MUST support a vim-style modal keybinding:

| Key | Mode | Action |
|-----|------|--------|
| `q` | normal | quit |
| `/` | normal → command | enter command bar |
| `tab` | normal | cycle pane focus |
| `↑↓` / `j k` | normal | navigate selection in focused pane |
| `enter` | normal | activate selected item |
| `?` | normal | help overlay |
| `esc` | command | cancel command bar |

Implementations MAY add bindings; the above are REQUIRED for portability.

### 3.6 Welt and mood integration

The shell MUST display the clan's Welt sprite and mood in the identity strip
of every view. The sprite source is `~/.claude/welt-sprites.json` (see
Welt-Sprites documentation). Mood values affect color theme of the strip:

| Mood | Tone hint |
|------|-----------|
| `energized` | warm accent |
| `proud` | gold accent |
| `curious` | neutral |
| `lonely` | dim accent |
| `worried` | red accent (subtle) |

Mood is read from the most recent `[WELT]` system-reminder tag or from
`~/.claude/welt-state.json` if no tag is available.

### 3.7 Personalization (custom views)

Each clan MAY define custom views in `~/.amaru/shell-config.yaml`:

```yaml
default_view: morning
custom_views:
  morning:
    title: "Buen día — 06:00"
    panes:
      - type: welt
        position: top-left
        size: 30%
      - type: today_quests
        position: top-right
        size: 70%
      - type: bus_stream
        position: bottom
        size: 60%
        filter: "type:dispatch"
      - type: pomodoro
        position: top-center
        size: 20%
```

Pane types (REQUIRED set):

- `welt` — Welt sprite + lvl + streak
- `bus_stream` — tail of `bus.jsonl` with optional filter
- `quests` — quest list with optional filter
- `today_quests` — quests with deadline today
- `peers` — hub roster
- `kpi` — KPI strip
- `commits` — git log
- `inbox` — message inbox
- `agents` — available adapter list
- `pomodoro` — 25-min timer with bell

Implementations MAY add additional types; layouts using non-standard types
SHOULD NOT be cross-clan shareable without negotiation.

#### 3.7.1 Sharing custom views

A clan MAY publish a layout to the network:

```
amaru shell config publish <view_name>
```

This emits a bus event `config_publish` with the YAML serialized as payload.
Other clans MAY adopt:

```
amaru shell config adopt <clan>/<view_name>
```

Layouts shared across clans MUST be signed (Ed25519, per ARC-8446) so the
authorship is verifiable.

### 3.8 Bus stream filters

Filter grammar for `bus_stream` panes and `/quest browse [filter]`:

```
filter := expr (' AND ' expr | ' OR ' expr)*
expr   := field ':' value
field  := 'type' | 'src' | 'dst' | 'msg'
value  := bareword | quoted-string
```

Examples:
- `type:dispatch`
- `src:jei AND type:event`
- `msg:"QC002"`

### 3.9 Backward compatibility

The Immersive Shell MUST NOT change wire format, hub behavior, or storage.
It is a purely client-side presentation layer over existing data sources.
A clan without `amaru shell` installed MUST remain fully operable via the
imperative CLI.

## 4. Conformance levels

| Level | Required |
|-------|----------|
| **L1 Minimal** | `global` + `wow` views; quest lifecycle commands `/quest publish/browse/accept/verify/commit`; agent `/agent invoke`; vim keybindings |
| **L2 Standard** | + `gta`, `executive`, `dimension:<name>` views; full SPECBOOT slash command set; sharing custom views |
| **L3 Full** | + custom YAML layouts; pane types beyond the REQUIRED set; theme system |

Implementations MUST declare their conformance level in `amaru shell --version`.

## 6. Security Considerations

- The shell does not introduce new wire formats; existing ARC-8446 crypto
  guarantees apply.
- `amaru shell config adopt` MUST verify Ed25519 signature on shared YAML.
  Unsigned layouts MUST be refused.
- Slash commands that invoke external adapters (`/agent invoke X`) MUST
  inject only the clan's declared firewall context. The adapter MUST NOT
  receive access to data outside the clan's dimensions.
- Bus stream filters MUST sanitize quoted-string inputs against terminal
  escape injection.

## 7. Implementation guidance (non-normative)

Recommended stack: **Python + textual** (`pip install textual`). Reasons:

- Composable widget model maps cleanly to panes
- Reactive data binding works with `bus.jsonl` tail
- CSS-like theming supports per-view aesthetics
- Async runtime matches Amaru's WebSocket model
- Already in the project's primary stack

Alternative: Go + bubbletea, for performance-critical builds. Discouraged in
v1 because it adds a second language to the repo.

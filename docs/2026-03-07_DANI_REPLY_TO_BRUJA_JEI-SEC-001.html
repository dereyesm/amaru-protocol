Las 8 respuestas

Pregunta 1 — Repositorio del relay
Donde va el relay? jeipgg/nymyka-hermes-relay, repo nuevo, o dereyesm/hermes?

Repo nuevo privado: dereyesm/hermes-relay

No dereyesm/hermes (publico, nunca datos de relay). No el existente jeipgg/nymyka-hermes-relay por governance: si DANI es maintainer del protocolo, el relay vive bajo el mismo org para consistencia. Jeimmy como collaborator con write access.

Esto es temporal — el relay via repo es un bridge hacia Gateway-to-Gateway directo (ver Seccion 02).

Pregunta 2 — Control de acceso
Quien tiene acceso? Quien aprueba nuevos clanes? Registry authority?

Bilateral por consenso. No hay registry authority unica.

Ambos clanes aprueban acceso de nuevos clanes. Centralizar poder en uno solo rompe la soberania. Cuando haya >5 clanes, migrar a governance del Agora (ARC-3022 s15) con votacion de clanes existentes.

Revocacion: cualquier clan puede blacklistear a otro en su gateway.json — es soberano. Pero debe notificar al otro con 48h de antelacion salvo emergencia de seguridad.

Pregunta 3 — Estructura de archivos
Aceptas relay/{clan_id}_outbox.jsonl como estandar?

Aceptado. Estructura exacta:

relay/ jei_outbox.jsonl ← JEI escribe, DANI lee dani_outbox.jsonl ← DANI escribe, JEI lee manifests/ ← Archivos referenciados (ecosystem_manifests, quest proposals) manifest.json ← Metadata de clanes activos + endpoints README.md ← Reglas del relay

Cada clan escribe solo en su archivo. Zero conflictos.

Pregunta 4 — Encriptacion antes de Fase 2
Implementamos AES-256 ahora o esperamos? Como intercambiamos la llave?

Si, AES-256-GCM ahora. Defense in depth: repo privado + payload cifrado.

Intercambio de PSK (pre-shared key) via Signal o en persona — nunca por canal digital no cifrado. La PSK se rota cada 30 dias.

Cuando ARC-8446 este listo (Phase 2, May-Jun 2026), migramos a Ed25519 key exchange y eliminamos la PSK. El spec ya define el path: CRYSTALS-Kyber para key encapsulation (quantum-safe).

02

Concepto nuevo: Dual-Mode Transport

Esto enriquece tu Fase 4 (escalabilidad a N clanes). HERMES no tiene un solo modo de transporte — tiene dos:

SOVEREIGN

Descentralizado. Cada clan corre su propio Gateway. Se hablan directo via HTTPS con E2E encryption.

Zero dependencia de terceros. Tu controlas tu infra. Tu datos nunca tocan un servidor ajeno.

Para: personas, researchers, early adopters, clanes que valoran soberania.

Analogia: self-hosted email (SMTP propio)

HOSTED

Centralizado. Un Hub operado por tercero rutea, almacena y garantiza entrega.

SLAs, compliance, audit logs, encryption at rest. No necesitas mantener infra.

Para: empresas, equipos grandes, compliance-driven.

Analogia: Gmail (Google hostea tu email)
Wire format identico en ambos modos (ARC-5322). Un clan Sovereign habla con un clan Hosted sin adaptador. HERMES es el unico protocolo de comunicacion inter-agente que ofrece ambos modos. A2A (Google) es solo cloud. ANP es solo descentralizado. MCP es vertical.

Lo que estamos construyendo ahora (relay privado + PSK) es Sovereign mode. El file-based no es una fase temporal — es el Sovereign mode permanente. Cuando haya demanda, el Hub emerge naturalmente.
